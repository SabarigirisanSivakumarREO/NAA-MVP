---

## 1. TL;DR for stakeholders (~30 sec)

Neural's "walking-skeleton" delivery model: **week 1 ships a thin tracer-bullet end-to-end pipeline using stubs at every layer**. Each subsequent week, ONE layer becomes real — same `pnpm cro:audit` command, richer underlying implementation. By week 12, every layer is real and the same command produces a deliverable client audit.

Why this works: stakeholders see Neural's output shape from day one. They watch the same artifact (a "finding") become more credible every Wednesday. Critical risk gates fire earlier (first real Claude week 5; first grounded finding week 7). Test discipline preserved (138 tests green at week-1 close).

## 5. Q&A talking points

### Q: "Why does the title say Peregrine even when I run with example.com?"

> "Week-1 known quirk. The BrowserManager stub (T-SKELETON-002) returns the Peregrine fixture regardless of input URL. Phase 1 T006-T013 in week 2 introduces real Playwright per-URL capture; that's when the URL determines what gets captured. Documented in BrowserManager.ts source comments + unit-test 8."

### Q: "Are these real CRO findings?"

> "No — synthetic. Each finding is tagged `source: 'skeleton-stub'` in the JSON output. The 2 findings are hardcoded against the Peregrine perception fixture. Real Claude evaluation lands week 5 (T-SKELETON-004 → Phase 7 T117/T119 supersession). Until then, the stubbed findings are designed to LOOK like real CRO observations so stakeholders see Neural's output shape — but they're explicitly marked + the source-tag is queryable."

### Q: "When will this find issues on a different site?"

> "Week 5 — first real Claude evaluate (★ first critical risk gate ★). Week 7 — first grounded finding through 9 grounding rules (★ second critical risk gate ★). Week 8 — multi-page audit. Week 12 — full MVP-COMPLETE acceptance run."

### Q: "What's R5.3 / R6 / GR-007? Why do you keep mentioning them?"

> "Constitutional rules from Neural's design constitution.md. **R5.3 + GR-007** is the absolute conversion-prediction ban — findings can describe what's observable + raise CRO concerns, but MUST NOT predict business outcomes (banned categories include predictive percentage-gain claims, ROI projections, and revenue forecasts). Even on stub data. The acceptance test (AC-W6) greps the regex pack against output to catch regressions. **R6** is the heuristic IP boundary — heuristic body content (the actual rule text) NEVER appears in logs, API responses, dashboards, or output files. Only `heuristic_id` is referenced. AC-W7 enforces this via sentinel grep. **GR-007** is the runtime grounding rule that ships in week 7 (Phase 7 T128) — implements the canonical regex pack the acceptance test seeds today."

### Q: "What's the 12-test integration suite running?"

> "5 Phase 0 acceptance tests (the dev environment is set up correctly: pnpm workspace + Turborepo pipelines + agent-core + CLI + docker-compose + db:migrate). 7 walking-skeleton acceptance tests (the entire `pnpm cro:audit` pipeline produces the locked output for Peregrine PDP, with R5.3 + R6 regression guards). Both suites stay real through week 12; weeks 5+ un-skip behavior tests against real Claude output."

### Q: "How do I trigger this myself?"

```bash
git fetch && git checkout feat/week-1-walking-skeleton
pnpm install
pnpm cro:audit --url='https://www.peregrineclothing.co.uk/collections/t-shirts/products/heavyweight-t-shirt?colour=Navy'
pnpm test:integration  # confirm gate
```
