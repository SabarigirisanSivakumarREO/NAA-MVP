# Neural MVP — Testing Strategy

> **Summary (~80 tokens — agent reads this first):** Test-first discipline (Constitution R3), stack (Vitest unit + Playwright integration + acceptance), coverage targets per module (90% on grounding, 80% on scoring, 70% on adapters, overall ≥ 70%), phase exit criteria from Spec Kit tasks.md, real-LLM policy (mock in unit/integration, real Claude in acceptance), per-component conformance matrix (CI-gated on every PR). Appendix B has full conformance test templates.

### 9.1 Philosophy

- **Test-first** (Constitution R3.1): write the failing test, see it fail, then implement
- **Never disable a failing test** (R3.3): fix the implementation
- **Every task has a smoke test** (R3.2) — defined in Spec Kit `tasks.md` (generated by CLI)
- **Findings-as-hypotheses applies to tests too:** tests assert concrete behavior, not opinions

### 9.2 Stack

| Level | Tool | Location | Purpose |
|---|---|---|---|
| Unit | Vitest | `packages/*/tests/unit/` | Function-level, isolated, <5s suite |
| Integration | Vitest + test DB + Playwright browser | `packages/*/tests/integration/` | Cross-module; real Postgres; real Chromium |
| Acceptance | Playwright Test | `tests/acceptance/` | Full CLI audit on 3 sites (example.com, amazon.in, Shopify demo) |
| Smoke (per task) | Inline in task description | Varies | Quick verification |

### 9.3 Coverage targets

| Path | Target | Rationale |
|---|---|---|
| `packages/agent-core/src/analysis/grounding/` | ≥ 90% | Hallucination filter; regressions are quality failures |
| `packages/agent-core/src/analysis/scoring/` | ≥ 80% | Deterministic 4D; easy to test; high downstream impact |
| `packages/agent-core/src/adapters/` | ≥ 70% | External boundaries; runtime failures expensive |
| `packages/agent-core/src/orchestration/` | ≥ 60% | Integration-heavy; acceptance suite catches rest |
| `packages/agent-core/src/browser-runtime/` | ≥ 50% | Real browser in integration catches most |
| `apps/dashboard/` | ≥ 40% | Smoke-tested in acceptance |
| Overall `packages/agent-core/` | **≥ 70%** | |

### 9.4 Phase exit criteria

Each phase in Spec Kit-generated `tasks.md` has exit criteria. A phase is "done" only when all pass. Final MVP exit: all 3 acceptance tests pass + 1 real client audit shippable.

### 9.5 Real-LLM policy

- Acceptance tests call real Claude Sonnet 4 (budget: ≤ $15 per acceptance run)
- Unit + integration: may mock LLM responses
- MVP: do NOT mock Claude in acceptance tests — catch prompt regressions against real model

### 9.6 Conformance test suite

Every critical component has a conformance test that verifies it meets its spec requirement, not just "does it run". Claude Code must run the conformance suite before declaring a task complete.

**Command:** `pnpm test:conformance` — runs the per-component conformance matrix below. See PRD Appendix B (or `docs/specs/mvp/templates/conformance-*.template.test.ts`) for full sample test cases.

**Conformance matrix (critical components):**

| Component | Conformance check | Expected behavior | Sample spec requirement |
|---|---|---|---|
| GR-001 (element exists) | Given a finding citing `ctas[5]` and a perception with `ctas.length === 3` → REJECT | Rejection with rule_id=GR-001, reason="element not found" | REQ-GROUND-001 |
| GR-007 (no conversion predictions) | Given a finding recommendation `"this will increase conversion by 15%"` → REJECT | Rejection with rule_id=GR-007, pattern match logged | R5.3 + REQ-GROUND-007 |
| GR-012 (benchmark validation) | Finding claims 14 fields against benchmark `6-8` (threshold_critical: 15) → PASS at `severity: "high"` | Accept; measurement cited | REQ-GROUND-012 |
| GR-012 (benchmark hallucination) | Finding claims 30 fields against benchmark `6-8` but actual form has 5 → REJECT | Actual-vs-claimed deviation > ±20% | REQ-GROUND-012 |
| Self-critique separation | Evaluate LLM + Self-critique LLM must have different system prompts | Snapshot test on prompt hashes | R5.6 |
| Temperature guard | Calling LLMAdapter.invoke({nodeName: "evaluate", temperature: 0.7}) → throws | Runtime Error: "temperature must be 0 for evaluate" | R10 |
| 2-stage heuristic filter | 30 heuristics → Stage 1 (ecommerce) → ~20 → Stage 2 (checkout) → 15-20 | Counts in expected range | §09.6 |
| Warm-up mode | New client (audits_completed=0) → publish any finding → finding.status = "held" | Not auto-published | §24.4 |
| AnalyzePerception schema | `page_analyze` output passes Zod validation on 3 fixture pages | Zod parse returns no errors | §07.9 |
| `page_analyze` single-call invariant | Playwright trace shows exactly 1 `page.evaluate()` call per analysis | Count === 1 | REQ-TOOL-PA-001 |
| Reproducibility | Run audit twice on same URL + same snapshot → finding overlap ≥ 90% | Jaccard similarity ≥ 0.9 | §25 |
| Annotate renders box + pin + ID | Output PNG has readable finding_id at correct bounding box | Pixel-level visual snapshot | §07.8 |
| Append-only tables | `UPDATE rejected_findings SET ...` → DB error | Constraint or trigger blocks | R7.4 |
| LLMAdapter contract | Mock returns malformed JSON 3× → adapter throws structured error after retry; logs to `llm_call_log` atomically | `LLMUnavailableError` raised; budget impact recorded; never silent | R9 + R14.1 + R14.5 |
| StorageAdapter contract | DB connection drops mid-transaction → adapter rolls back cleanly; RLS context not bleeding | Transaction aborts; no partial writes; `app.client_id` reset on next acquire | R7.1 + R7.2 |
| ScreenshotStorage contract | R2 upload fails in dev mode → falls back to LocalDisk; in prod → errors to caller | Screenshot persists or fails loudly; never silent loss | R7.3 + §7.5 |
| BrowserEngine contract | Playwright `page.evaluate()` throws → adapter retries once, then escalates per R23 | Session refreshed; page retry per kill criteria; never crashes the audit | R9 + R23 |
| Migration schema drift | After `pnpm db:migrate`, `pnpm db:introspect` output matches Drizzle schema definition | No drift; CI fails on mismatch | R7.1 |

**Policy:**
- Conformance tests run in CI **on every PR** (unlike acceptance tests which run nightly on main)
- A PR cannot merge if any conformance test fails
- New grounding rule or scoring component requires a new conformance test in the same PR
- Conformance tests live at `packages/agent-core/tests/conformance/<component>.test.ts`

**Claude Code usage rule:** After completing a task, run `pnpm test:conformance -- <component>` for every component the task touched. If any fails, debug before declaring complete (PRD §10.6 self-verification).

### 9.7 Regression policy (clarification)

There is **no separate `tests/regression/` directory**. Regression coverage is the **union of**:

- All unit tests (`packages/*/tests/unit/`)
- All integration tests (`packages/*/tests/integration/`)
- All conformance tests (`packages/agent-core/tests/conformance/`) — see §9.6 matrix
- All acceptance tests (`tests/acceptance/`)

These suites run **CI-gated on every PR**. **Constitution R23.2 kill criterion enforces "any previously-passing test breaks → STOP."** A failed regression is a kill-criterion trigger, not a gate-pass-with-warning.

The Spec Kit `tasks-template.md` does NOT generate a separate regression test category. The regression discipline is enforced at the kill-criterion level on every test run; tasks that introduce regressions trigger the R23.4 protocol (snapshot WIP → log → escalate, never silently retry).

### 9.8 Dependency / adapter contract test policy

Per Constitution R9, every external dependency goes through an adapter (`LLMAdapter`, `StorageAdapter`, `ScreenshotStorage`, `BrowserEngine`, `JobScheduler`, `EventBus`, `HeuristicLoader`, `NotificationAdapter`, `AuthProvider`, `DiscoveryStrategy`, `StreamEmitter`). Each adapter MUST have a **contract test** that verifies:

1. The adapter exposes the required interface methods (TypeScript types are not enough — runtime test required)
2. The adapter handles vendor failure modes (timeout, rate limit, malformed response, network drop) without leaking vendor-specific errors to callers
3. The adapter respects budget gates (LLMAdapter only — pre-call budget estimate via `getTokenCount()`)
4. The adapter logs to `llm_call_log` atomically per call (LLMAdapter only — R14.1)
5. The adapter never sets `temperature > 0` on `evaluate` / `self_critique` / `evaluate_interactive` (LLMAdapter only — R10 via `TemperatureGuard`)

Adapter contract tests live in `packages/agent-core/tests/conformance/<adapter-name>-contract.test.ts`. Templates at `docs/specs/mvp/templates/conformance-test-templates.md` §6.

**Migration validation tests:** the Drizzle schema must match the actual Postgres tables after every migration. After `pnpm db:migrate`, a test runs `pnpm db:introspect` and compares against the schema definition. Drift fails CI immediately. Template at `docs/specs/mvp/templates/conformance-test-templates.md` §7.

**When mandatory:**

- **Phase 0 (Setup):** all 11 adapter interfaces scaffolded → 11 adapter contract test stubs created (each can fail with `not implemented` until the adapter lands in later phases — that's acceptable for stubs)
- **Phase 1 onward:** when an adapter's concrete implementation lands, the contract test must pass before the phase exit gate
- **Migration tests:** every Phase that adds a Drizzle migration includes a matching migration validation test (Phase 0 baseline schema; Phase 4 cost tables; Phase 9 dashboard tables)
