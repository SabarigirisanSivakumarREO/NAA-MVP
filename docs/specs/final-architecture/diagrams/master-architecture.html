L3 ANALYSIS ENGINE AI-powered evaluation
5-Step Analysis Pipeline
PERCEIVE
~200ms
Collects AX-tree snapshot, screenshots, metadata, and network data from the browser agent. Pure data gathering, no LLM calls.
▶
EVALUATE
~3–5s
LLM evaluates perceived data against the filtered heuristic set (15-20 per page). Produces draft findings with severity, location, and initial evidence.
▶
SELF-CRITIQUE
~2–3s
Second LLM pass reviews draft findings for hallucinations, weak evidence, false positives. Applies Evidence Grounder rules GR-001 through GR-011. Drops or downgrades findings that fail grounding.
▶
GROUND
~100ms
Deterministic validation of evidence references. Verifies CSS selectors resolve, screenshots match claims, and all REQ-IDs are valid. No LLM involved.
▶
ANNOTATE
~500ms
Generates annotated screenshots with bounding boxes, finding markers, and severity color coding. Uploads to R2 storage.
Analysis Components
▼
REQ-HEUR-* Heuristic Evaluator
100 heuristics across 3 tiers (Universal, Conditional, Advanced)
Evaluates page data against the filtered heuristic subset. Each heuristic defines: criteria, evidence requirements, severity mapping, and industry applicability. Tier 1 (Universal) always runs; Tier 2/3 are context-dependent.
▼
GR-001..GR-011 Evidence Grounder
11 grounding rules to eliminate hallucinations
11 deterministic rules (GR-001 through GR-011) that validate every finding's evidence chain. Checks include: selector existence, screenshot region verification, metric substantiation, cross-reference consistency, and temporal validity.
▼
Cross-Page Consistency
Detects contradictions and patterns across pages
Compares findings across all pages in an audit to detect contradictions, identify site-wide patterns, and deduplicate similar issues. Uses pgvector semantic similarity for finding clustering.
▼
Competitor Comparator
Benchmarks against competitor implementations
Optionally compares audit target against competitor URLs. Identifies relative strengths/weaknesses and industry-standard patterns the target may be missing.
▼
REQ-REVIEW-* Review Gate
Human-in-the-loop quality checkpoint
All AI-generated findings pass through human review before client delivery. Consultants approve, reject, or edit findings. Rejection reasons feed back into heuristic calibration. Enforces minimum quality bar.

REQ-ANALYZE-NODE-005
Three-Layer Hallucination Filter
L1
Chain-of-Thought (Step 2) LLM reasoning constraint
Catches: unfounded claims during generation
Filter rate: ~50%
L2
Self-Critique (Step 3) LLM self-review
Catches: logical inconsistencies, severity errors
Filter rate: ~30% of remaining
L3
Evidence Ground (Step 4) Deterministic code
Catches: hallucinated elements, wrong measurements
Filter rate: ~95% of remaining

Self-Critique Prompt (Step 3) — SEPARATE LLM call, different persona (SD-07)
SYSTEM PROMPT (~400 tokens)
"You are a senior CRO quality reviewer. CHALLENGE each finding with rigorous scrutiny."
5 checks per finding:
1. VERIFY ELEMENT: Does referenced data exist in page data? Contradicts → REJECT
2. CHECK SEVERITY: critical needs MEASURABLE evidence, high needs element-level. Inflated → DOWNGRADE
3. CHECK LOGIC: Does conclusion follow from observation? "4 fields = too many" is bad logic → REJECT
4. CHECK CONTEXT: 10-field form is fine for mortgage, bad for newsletter. Context ignored → REVISE
5. CHECK DUPLICATES: Same issue reported twice? → MERGE
Output per finding: { finding_index, heuristic_id, verdict: KEEP|REVISE|DOWNGRADE|REJECT, reason, revised_finding?, new_severity? }
USER MESSAGE (~6,000 tokens — findings + page data, NO heuristics)
ORIGINAL PAGE DATA:
{{analyze_perception | json}}
---
FINDINGS TO REVIEW (violations + needs_review only — passes filtered out):
{{raw_findings.filter(f => f.status !== "pass") | json}}
---
"Review each finding. Be harsh — it's better to reject a valid finding than to let a hallucinated one through."
KEEP
Finding is solid, evidence checks out
REVISE
Logic/context needs fix, revised_finding provided
DOWNGRADE
Severity inflated, new_severity assigned
REJECT
Hallucinated, illogical, or duplicate → removed

Why Pass 2 exists
Structured heuristics catch known patterns. But a human CRO consultant also notices things no formal rule covers — awkward micro-copy, confusing visual hierarchy, missing urgency cues, or trust gaps specific to the industry. Pass 2 captures these "consultant instinct" observations. They're always Tier 3 (consultant-reviewed) and feed the Learning Service — approved observations crystallize into new heuristics over time. The system literally gets smarter from its own discoveries.
