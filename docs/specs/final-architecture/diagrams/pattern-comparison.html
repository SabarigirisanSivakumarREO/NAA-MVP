3. Pattern B — Subway / metro map

Each flow type is a colored "line" routed through stations (components). Trace one line at a time. Mock at 16 nodes:

Strength: The eye traces one colored line at a time. Most beautiful single-image option. Familiar mental model. Wall-poster shareable.
Weakness: Hand-laid-out — auto-layout doesn't produce subway-map quality. Each new node = manual SVG editing. High maintenance cost.
At 145 nodes: Doable but ~2 weeks of layout tuning. The eye still works because lines stay isolated by color. Best with 5–6 lines max — would need to consolidate flow types.

Strength: Text-driven (easy to maintain in git). Auto-layout. Subgraphs visually group concerns. Engineers love Mermaid.
Weakness: Auto-layout struggles past ~80 nodes — Mermaid starts overlapping. Click-to-drawer needs custom JS overlay on the SVG. Limited control over node coordinates.
At 145 nodes: Will produce a long-but-readable diagram if split into 4 Mermaid blocks (one per Part). Combined with the C4 overview pattern, this is a strong contender.

5. Pattern D — Force-directed graph with collapsible Part groups (Cytoscape.js)

Each Part is a parent node. Use the buttons to expand / collapse Parts. Drag nodes around. Click any node to log details.

Strength: User controls density via expand/collapse. Scales to 145+ without showing all at once. Drag-to-rearrange. Strong drill-down feel.
Weakness: Force-directed layout can be ugly on first render (jiggly). New library dep. Coordinates aren't deterministic — diagrams differ between sessions.
At 145 nodes: Works well IF Parts stay collapsed by default. Expanding all 4 = mild hairball but recoverable via re-layout. Better than baseline.

6. Pattern E — Dependency Structure Matrix (DSM)

Rows = source, columns = target. Cell color = flow type. Zero edge crossings — every relationship is a single cell.

control data analysis infra safety
Strength: Shows ALL relationships at once with zero edge crossings. Industry-standard for large-scale dependency analysis. Clean at any scale.
Weakness: Alien at first glance. Doesn't tell a flow narrative — you can't "read" a story from it. Each cell needs a tooltip for context.
At 145 nodes: Becomes a 145×145 grid. Still readable IF Parts/layers are clustered into block-diagonal groups (you'd see "this Part has dense internal coupling, sparse external"). Useful for architects, less so for narrative.

My pick: Pattern 2 (C4 drill-down) as the primary engineering view, with Pattern 4 (Mermaid) embedded inside each Part-detail view for the flow diagrams. Engineers and AI agents get a 4-Part overview that fits on one screen, drill into the Part they care about, and see a clean Mermaid graph for that Part's components — clicking any component opens the existing drawer. Maintains everything you've already built (drawer schema, dependencies, internal_architecture) and avoids the hairball entirely.

Alternative if you want one beautiful artifact: Pattern 3 (subway map) — but I'd budget a full session for layout tuning.
