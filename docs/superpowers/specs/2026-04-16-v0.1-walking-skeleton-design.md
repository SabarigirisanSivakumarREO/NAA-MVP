# Neural v0.1 — Walking Skeleton Design Spec

> **Status:** Design approved by user (via brainstorming session 2026-04-16). Ready for implementation plan.
> **Scope:** ~13 tasks. 1-2 weeks. Proves pipeline end-to-end on 1 page + 1 heuristic.
> **Purpose:** Get evidence before committing to v1.0 MVP (60-80 tasks). Not a pilot-ready product.
> **Predecessor docs:** `docs/PROJECT_BRIEF.md`, `docs/specs/mvp/tasks-v2.md`, `docs/master-architecture-checklist.md`

---

## 1. Goal & Non-Goals

### What v0.1 proves

Given a URL and a page type, the system produces:
- A JSON file of grounded findings (hypotheses with evidence)
- An annotated PNG showing each finding boxed on the screenshot with a finding ID
- A debug log of LLM calls + costs

Concretely, after v0.1 ships, we can answer **with evidence**:

1. Does the browser-perception-LLM-grounding pipeline actually produce useful findings on real web pages?
2. How often does evidence grounding reject hallucinated findings?
3. What is the real cost per page in production? (vs projected $0.35)
4. Are benchmark-driven findings better than benchmark-less findings?
5. What's the false-positive rate on real sites?

### What v0.1 is not

- Not a pilot product. Consultants cannot use it to audit real clients.
- Not production infrastructure. No DB, no auth, no UI, no emails.
- Not multi-heuristic. 1 heuristic (`CHECKOUT-FORM-FIELDS`) is the scope.
- Not multi-page. Takes a single URL. Multi-page loop comes in v0.2.
- Not stealth/anti-bot hardened. Default Playwright only.
- Not a full browser agent. ~8 MCP tools, hardcoded flow. No reason/reflect nodes.
- Not PDF report. JSON + PNG are the deliverables.

### Success criteria

v0.1 is "done" when:
- [ ] `pnpm proof --url <url> --page-type checkout` runs without errors on 5 real checkout pages
- [ ] At least 1 page produces at least 1 grounded finding with valid evidence (element exists, bounding box correct)
- [ ] At least 1 page produces at least 1 finding that grounding REJECTS (validates hallucination filter works)
- [ ] Annotated screenshot shows the finding location with a visible box + ID
- [ ] Total runtime < 60 seconds per page
- [ ] Total cost < $0.50 per page (Claude Sonnet 4, temp=0)
- [ ] All smoke tests pass (unit + integration)

## 2. Architecture

### 2.1 Pipeline (linear, no graph)

```
CLI entrypoint
  │
  ▼
BrowserManager.open()
  │
  ▼
navigate(url)
  │
  ▼
MutationMonitor.waitForStability()
  │
  ▼
ScreenshotExtractor (viewport + fullpage)
  │
  ▼
page_analyze() ──► AnalyzePerception { ctas, forms, headings, textContent }
  │
  ▼
evaluate() ──► [RawFinding]                   (LLM call 1: Claude Sonnet 4, temp=0)
  │
  ▼
self_critique() ──► [ReviewedFinding]         (LLM call 2: different persona, temp=0)
  │
  ▼
ground() ──► [GroundedFinding] + [RejectedFinding]   (deterministic, 3 rules)
  │
  ▼
annotate(viewport_screenshot, findings) ──► annotated PNG
  │
  ▼
write outputs to ./output/run-<timestamp>/
```

No LangGraph.js in v0.1. Linear async function composition. Graph topology is added in v0.2 when we introduce the page loop.

### 2.2 Deliberate simplifications vs master plan

| Master plan (v2.2a) | v0.1 | Rationale |
|---|---|---|
| 23 browse MCP tools | ~8 direct function calls (navigate, waitForStability, screenshot, page_analyze, fullpage_screenshot) | No MCP wrapping yet. Wrapping adds surface without pipeline value. |
| Full AnalyzePerception (headingHierarchy, landmarks, semanticHTML, textContent, ctas, forms, trustSignals, layout, images, navigation, performance) | Minimal: `{ ctas, forms, headings, textContent, metadata }` | Enough for 1 heuristic. Full extraction added when heuristics need it. |
| 100 heuristics, Baymard/Nielsen/Cialdini, AES-256-GCM encrypted, 2-stage filter | 1 heuristic in a TS constant: `CHECKOUT_FORM_FIELDS` with quantitative benchmark | Prove the schema + injection + grounding; scale to 30 in v0.2. |
| 12 grounding rules (GR-001..GR-012) | 3 rules: GR-001 (element exists), GR-007 (no conversion predictions), GR-008 (data_point references real section) | These 3 cover the most common hallucination modes. Others added with heuristics that need them. |
| LangGraph.js state graph + checkpointer | Linear async functions | Graph machinery adds weeks. Defer to v0.2. |
| Postgres 25+ tables + RLS + Drizzle | JSON files on disk (`./output/run-<timestamp>/`) | DB schema is ~15 tasks. Skip until we have multi-run queries. |
| Cloudflare R2 | Local disk | Same reason. |
| 2-stage heuristic filter (business_type → page_type) | 1 heuristic, page_type=checkout hardcoded | Nothing to filter. |
| Self-critique with full 5-check prompt | Simplified: 3 checks (element existence, severity proportionality, logical coherence) | Prove the separate-persona pattern; expand later. |
| Sharp annotation with severity colors, overlap avoidance, connection lines | Sharp annotation with: colored box at bounding box, numbered pin, basic overlap nudge | Basic visual correctness is enough. |
| Stealth plugin, ghost-cursor, fingerprint rotation | Default Playwright | We aren't loading authenticated sites or provoking bot detection. |
| Reproducibility snapshot | `audit_run.git_commit_hash` + `audit_run.model_name` in output JSON | Good-enough reproducibility for v0.1. |
| Temperature guard adapter | Pass `temperature: 0` explicitly in each LLMAdapter call + runtime assertion | Simpler than adapter-level guard. |
| Cost tracker with llm_call_log table | `cost.txt` with a one-liner per call | Log to disk, not DB. |
| MCP server, session management, tool registry | Direct function calls | We reintroduce MCP in v0.2 when we add the second heuristic. |

## 4. Component Specifications

### 4.1 BrowserManager

**Responsibility:** Wraps Playwright. Opens a Chromium context, navigates, returns `page` handle.

**Interface:**

```typescript
interface BrowserManager {
  open(): Promise<void>;
  navigate(url: string, opts?: { waitUntil?: "load" | "domcontentloaded" }): Promise<void>;
  getPage(): Page;              // Playwright Page handle (v0.1 leak; formalized in v0.2)
  close(): Promise<void>;
}
```

**Behavior:**
- `chromium.launch({ headless: true })` — no stealth, no fingerprint rotation
- Viewport: 1440×900 (desktop only)
- User-agent: default
- Timeout on navigate: 30s
- Throws on navigation failure (no retry yet)

### 4.2 MutationMonitor

**Responsibility:** Wait for DOM to stop mutating (proxy for "page loaded").

**Interface:**

```typescript
function waitForStability(page: Page, opts?: { timeoutMs?: number; settleMs?: number }): Promise<void>;
```

**Behavior:**
- Inject `MutationObserver` via `page.evaluate()`
- Consider page "stable" after no mutations for `settleMs` (default 1500ms) within `timeoutMs` (default 10s)
- If timeout, proceed anyway (non-fatal — the page might just be a streaming app)

### 4.3 ScreenshotExtractor

**Responsibility:** Capture viewport + fullpage JPEGs.

**Interface:**

```typescript
function captureViewport(page: Page): Promise<Buffer>;    // JPEG
function captureFullpage(page: Page, maxHeight?: number): Promise<Buffer>;   // JPEG, max 15000px
```

**Behavior:**
- Quality 80 for viewport, 70 for fullpage
- Viewport < 500KB, fullpage < 2MB typically
- Returns Buffer; caller writes to disk

### 4.4 `page_analyze`

**Responsibility:** Single `page.evaluate()` call returning minimal AnalyzePerception.

**Interface:**

```typescript
async function pageAnalyze(page: Page): Promise<AnalyzePerception>;
```

**Behavior (runs in browser context):**

```javascript
() => {
  const viewport = { width: window.innerWidth, height: window.innerHeight };

  // CTAs: any <button>, <a> with button-like styling, or input[type="submit"]
  // above-the-fold = getBoundingClientRect().top < window.innerHeight
  // Contrast ratio: compute getComputedStyle().color vs parent backgroundColor

  // Forms: every <form>, collect fields with name/type/required/hasLabel
  // hasLabel = document.querySelector(`label[for="${field.id}"]`) !== null OR closest('label')

  // Headings: document.querySelectorAll('h1,h2,h3,h4,h5,h6')

  // textContent: innerText of main content area, split into paragraphs

  return { metadata: {...}, ctas: [...], forms: [...], headings: [...], textContent: {...} };
};
```

**Validation:** Output validated against `AnalyzePerceptionSchema` before returning.

### 4.6 LLMAdapter

**Responsibility:** Wrap `@anthropic-ai/sdk`. Enforce temperature=0. Return structured output.

**Interface:**

```typescript
interface LLMResponse<T> {
  parsed: T;                             // Zod-validated
  raw: string;                           // raw LLM text for debug log
  usage: { input_tokens: number; output_tokens: number };
  cost_usd: number;
  duration_ms: number;
  model: string;
}

interface LLMAdapter {
  invoke<T>(args: {
    system: string;
    user: string;
    schema: z.ZodSchema<T>;
    nodeName: "evaluate" | "self_critique";   // for temperature guard
  }): Promise<LLMResponse<T>>;
}
```

**Behavior:**
- Runtime assertion: if `nodeName in ["evaluate", "self_critique"]` and `temperature !== 0`, throw
- Model: `claude-sonnet-4-20260301` (hardcoded)
- Max tokens: 4000
- Cost computed from `MODEL_PRICING["claude-sonnet-4-20260301"]` constant
- Zod parse; if parse fails, throw (no retry in v0.1)

### 4.7 `evaluate` (LLM call 1)

**Responsibility:** Send perception + 1 heuristic to LLM. Get `RawFinding[]`.

**Prompt structure:**

**System prompt (~300 words):**
> You are a CRO (conversion rate optimization) analyst reviewing a web page. Your job is to evaluate the page against a specific heuristic and produce findings.
>
> A finding is a HYPOTHESIS about a potential conversion issue. Findings must be:
> 1. Grounded in observable evidence (specific elements on the page)
> 2. Reference a specific data_point in the page perception
> 3. Include a measurement or concrete observation
> 4. Cite the heuristic benchmark
>
> DO NOT predict conversion impact ("this will increase conversions by 20%"). DO NOT speculate about user psychology beyond what the heuristic says. DO NOT produce findings about elements that don't exist.
>
> Output ONLY valid JSON matching the schema. No commentary.

**User message:**
```
# HEURISTIC TO EVALUATE

[heuristic JSON with benchmark]

# PAGE PERCEPTION

URL: [url]
Page type: checkout
Business type: ecommerce
Forms: [JSON of forms]
CTAs: [JSON of ctas]
Headings: [JSON of headings]
Word count: [n]

# TASK

Evaluate whether this page violates the heuristic above. If yes, produce a finding. If no, produce a pass finding. If ambiguous, produce a needs_review finding. Cite the benchmark in your assessment.

# OUTPUT SCHEMA

[RawFindingSchema as JSON schema]
```

**Post-processing:** Parse + Zod validate. If parse fails, throw (v0.1 has no retry).

### 4.8 `self_critique` (LLM call 2)

**Responsibility:** A second LLM call with a different persona reviews the finding and issues a verdict.

**Prompt structure:**

**System prompt:**
> You are a SENIOR QUALITY REVIEWER checking the work of a junior CRO analyst. Your job is to spot hallucinations, unsupported claims, and inflated severity.
>
> For each finding provided, perform 3 checks:
> 1. ELEMENT EXISTS: Does the element_ref or selector actually appear in the perception data?
> 2. SEVERITY PROPORTIONALITY: Is the severity warranted by the evidence? If measurement is slightly off-benchmark, severity should be "low" or "medium", not "critical".
> 3. LOGICAL COHERENCE: Does the observation + assessment + recommendation form a coherent argument?
>
> Issue a verdict:
> - KEEP — all 3 checks pass
> - REVISE — minor issue; suggest a revision
> - DOWNGRADE — severity is too high; downgrade to the correct level
> - REJECT — at least one check fails fatally (element doesn't exist, logic is broken, etc.)

**User message:** perception + list of RawFindings + output schema (`ReviewedFinding[]`).

### 4.9 `ground` (deterministic, no LLM)

**Responsibility:** Run 3 grounding rules on each ReviewedFinding. Separate into GroundedFinding + RejectedFinding.

**Rules:**

- **GR-001 (element exists):** If `evidence.element_ref` or `evidence.selector` is not null, verify an element matching it exists in the perception data. How: traverse `AnalyzePerception` looking for a match by selector or ref. If not found → REJECT.
- **GR-007 (no conversion predictions):** Scan `observation`, `assessment`, `recommendation` for banned phrases: "increase conversion", "boost revenue", "%[\s]*(lift|increase|improvement)", "ROI of". If any match → REJECT.
- **GR-008 (data_point references real section):** `evidence.data_point` must be a valid dotted path into the perception. E.g., `forms[0].fieldCount` → check `perception.forms[0].fieldCount` exists. If not → REJECT.

**Implementation:** Each rule is a pure function `(finding, perception) => { pass: true } | { pass: false, reason: string }`. Composable into a pipeline.

### 4.10 `annotate`

**Responsibility:** Draw a colored box at each finding's bounding box on the viewport screenshot, with a numbered circle showing the finding_id.

**Approach:**
- Use Sharp to overlay SVG onto JPEG
- Build SVG string with:
  - `<rect>` at bounding box, stroke-width 3, stroke-color by severity (critical=red, high=orange, medium=yellow, low=blue)
  - `<circle>` + `<text>` at top-left of bounding box with finding ID number
- If two findings overlap, offset the second pin by 20px (basic nudge, no overlap avoidance graph)

**Interface:**

```typescript
function annotate(opts: {
  viewportJpeg: Buffer;
  viewportDimensions: { width: number; height: number };
  findings: GroundedFinding[];
}): Promise<Buffer>;      // annotated JPEG
```

### 4.11 CLI entrypoint

**Command:**
```bash
pnpm proof --url <url> --page-type checkout [--output ./output]
```

**Behavior:**
1. Parse flags (use `commander` or similar)
2. Generate `audit_run_id` (uuid)
3. Create output dir `./output/run-<ISO-timestamp>/`
4. Run pipeline
5. Write manifest.json, findings.json, rejected.json, viewport.jpg, viewport-annotated.jpg, fullpage.jpg, llm-calls.jsonl, cost.txt
6. Print summary to stdout:
   ```
   ✓ Audit complete
   Run ID: <uuid>
   Output: ./output/run-<timestamp>/
   Findings: 2 grounded, 1 rejected
   Cost: $0.047
   Duration: 24.3s
   ```
7. Exit 0 on success, 1 on failure

## 5. Testing Strategy

### 5.1 Unit tests (Vitest)

- `ground.test.ts` — each of 3 grounding rules with passing + failing cases
- `pageAnalyze.test.ts` — given a fixture HTML string, returns expected perception
- `annotate.test.ts` — given a fixture screenshot + findings, output PNG exists and is non-empty
- `llmAdapter.test.ts` — temperature=0 assertion throws on non-zero

### 5.2 Integration test

- `pipeline.test.ts` — feed a fixture HTML (saved Shopify checkout page) through the full pipeline with a mocked `LLMAdapter` that returns canned responses. Assert:
  - At least 1 RawFinding produced
  - Self-critique applies a verdict
  - Grounding either accepts or rejects
  - Annotated output PNG exists

### 5.3 Manual smoke tests

Run `pnpm proof` on 5 real checkout URLs (provided by CRO team). For each:
- Inspect `findings.json` — are findings reasonable? Evidence valid?
- Inspect `viewport-annotated.jpg` — are boxes on the right elements?
- Inspect `rejected.json` — did grounding catch anything real?
- Inspect `cost.txt` — what's the actual per-page cost?

Record results in `docs/v0.1/manual-smoke-results.md`.

### 5.4 No golden tests yet

Golden test infrastructure (§36 in master plan) is a v1.0 concern. In v0.1, manual inspection is the quality bar.

## 7. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Anthropic API unavailable | Low | High | v0.1 has no failover. Manual retry. Accept. |
| Playwright fails on some target sites (bot detection) | Medium | Medium | Pick test sites that don't have aggressive bot detection. Move stealth to v0.2 if needed. |
| LLM returns malformed JSON | Medium | Medium | Zod parse throws; v0.1 does not retry. Caller sees error. Accept for v0.1. |
| `page.evaluate()` throws on complex SPAs | Medium | Low | Test on static-ish pages first. Add shadow DOM handling in v0.2. |
| Bounding box coordinates wrong after viewport/zoom | Low | Medium | Viewport fixed at 1440×900 in v0.1. No zoom handling. |
| Self-critique rejects all findings | Low | Low | If happens, rerun without self-critique. Log the issue. |
| Real per-page cost > $0.50 | Low | Low (learning) | This is info we WANT from v0.1. Not a failure. |
| Grounding rejects nothing (no hallucinations caught) | Possible | Informational | Interesting signal. Means either heuristic is too safe or LLM is reliable on this task. |
| 5 test sites all have zero findings | Low | High | Indicates heuristic/prompt is broken. Iterate on prompt. |

## 8. Decision Checkpoint (End of v0.1)

After v0.1 ships and we run it on 5-10 real checkout pages, schedule a 1-hour review with these questions:

1. **Finding quality:** How many findings were accurate? How many were false positives that survived grounding?
2. **Grounding effectiveness:** What percentage of RawFindings were rejected? Which rule rejected them?
3. **Cost reality:** Actual $/page vs the projected $0.35? If 2x+ off, rethink architecture.
4. **Duration:** Actual seconds/page. Is it fast enough to scale to 20 pages?
5. **LLM reliability:** How often did Claude return malformed output?
6. **Benchmark value:** Did benchmark injection produce obviously better findings than no-benchmark?
7. **Annotation clarity:** Are the boxes useful? Is the finding_id discoverable on the image?

**Outcomes:**
- **Green (all good):** Proceed to v0.2 (5 heuristics + multi-page + Postgres).
- **Yellow (pipeline works but quality is marginal):** Iterate on prompt + self-critique + heuristic. Don't scale up yet.
- **Red (pipeline produces junk):** Rethink. Maybe we need state exploration sooner. Maybe the heuristic needs a different structure. Maybe the prompt needs fundamental rework.
